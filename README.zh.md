# Behavior-Driven Development (BDD)

[English](./README.md)

## 为什么需要 BDD？

在传统软件开发里，开发者脑子里想的往往是「我要实现这个功能模块」、「这个类要处理这些数据」。代码写完，功能跑通，但和产品经理、用户、业务方真正想要的「行为」对不上。测试写了一堆，测的却和业务真正关心的场景完全不是一回事——沟通成本高，反复返工。

BDD（行为驱动开发）就是为了解决这个问题：**我和别人对同一件事的理解，根本不在一个频道上。**

## 核心思想

> 软件开发应该围绕「这个系统在不同场景下应该表现出什么行为」来展开，而不是围绕「我要写什么代码」来展开。

BDD 把焦点从「技术实现」拉回到「可观察的行为」。

不是跟团队说「我们需要一个用户登录模块」，而是大家一起用业务语言描述：

> 当一个已注册的用户输入正确的用户名和密码时，系统应该让他成功登录并跳转到首页；  
> 当他输入错误的密码三次后，系统应该锁定账号并提示用户联系客服。

这不是技术文档，也不是代码，而是任何人都能读懂的「行为描述」。

## BDD 和 TDD 的关系

BDD 是 TDD 的演化版，把 TDD 的思路从「代码层面」提升到了「行为层面」：

| | TDD | BDD |
|---|---|---|
| 核心问题 | 这个函数的输入输出是否符合测试？ | 这个功能在真实场景里的行为是否符合业务预期？ |
| 测试粒度 | 单元测试为主 | 场景/行为为主 |
| 语言 | 代码 | 接近自然语言（Given-When-Then） |
| 受众 | 开发 | 产品、开发、测试都能读 |

BDD 依然先写「测试」，但这些测试是用行为场景描述的，测试名称用 `should ...` 或 `when ... then ...` 来表达意图。

## Given-When-Then

BDD 用三段式结构描述场景：

- **Given**（前提）：系统当前是什么状态
- **When**（操作）：用户/系统做了什么
- **Then**（结果）：系统应该表现出什么行为

```gherkin
Given 用户已注册且账号正常
When 用户输入错误密码三次
Then 账号被锁定，并提示联系客服
```

映射到代码：

```typescript
describe('用户登录', () => {
  describe('when 用户连续输入错误密码三次', () => {
    it('should 锁定账号并返回联系客服提示', async () => {
      ...
    });
  });
});
```

## BDD 的价值

1. **统一语言**：产品、开发、测试用同一套描述说话，消除理解偏差。
2. **活文档**：场景描述既是测试用例，也是永远不过时的需求文档。
3. **更早暴露认知差距**：在需求阶段就把行为描述清楚，三方对功能的理解不一致时立刻可见。

## 为什么在使用 Coding Agent 开发时，BDD 比 TDD 更好

TDD 是为脑子里装着业务意图的人类开发者设计的。Coding Agent 没有这个上下文——这个不对称性，让 BDD 在 AI 辅助开发流程中具有明显优势。

**1. Agent 实现的是 spec，不是意图**

把一个 TDD 单元测试交给 Agent，它会写出刚好能让这个测试通过的最小代码——仅此而已。如果测试是围绕实现细节写的，而不是围绕行为，Agent 产出的代码满足了测试，但偏离了真实需求。BDD 场景写在契约边界上，不给这种偏差留空间。

**2. 没有行为约束时，Agent 天然倾向于过度设计**

给 Agent 一个开放的 prompt，它会主动加 options、抽象层、没人要求的边界处理。一个失败的 BDD 场景是一道硬约束："只实现让这个可观察行为通过的代码。" TDD 单元测试没有这种约束力。

**3. Agent 写的 TDD 测试是自我验证的**

让 Agent 在实现之后补测试，它写出来的测试会匹配它自己写的代码——而不是真正的需求。这是经典的 TDD 陷阱，Agent 会稳定地掉进去。BDD 场景应该由人类（或协商后）在 Agent 写任何实现代码之前就确定下来。

**4. 场景不需要读代码就能 review**

`describe('当用户连续输入错误密码三次') → it('应该锁定账号')` 任何人都能验证对不对。Agent 写的充满 mock 的单元测试，需要理解实现才能判断它测的是不是正确的事。BDD 把 review 的成本保持在人类可以处理的规模。

**5. 契约层测试能在 Agent 重构时存活**

Agent 重构很激进。每次 Agent 重整内部结构，与内部方法绑定的单元测试就会挂掉——即使行为完全没变。写在公共契约边界上的 BDD 测试，在任何内部重构中都能保持绿色，让 Agent 可以自由重构，不会产生测试噪音。

**一句话总结：** TDD 要求开发者把意图装在脑子里，然后忠实地转化为测试。Agent 做不到这种转化。BDD 把意图提前外化为人类可读的场景——让 Agent 的任务明确无歧义，输出也变得可验证。
